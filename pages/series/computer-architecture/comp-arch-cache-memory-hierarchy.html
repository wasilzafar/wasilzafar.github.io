Introduction

Modern computers use a memory hierarchy to balance speed and capacity. Fast memory is expensive and limited; slower memory is cheap and abundant. Caches bridge this gap.

Series Context: This is Part 14 of 24 in the Computer Architecture & Operating Systems Mastery series. We transition from process management to memory systems.

Memory is organized in a pyramid: smaller, faster, more expensive memory at the top; larger, slower, cheaper memory at the bottom.

Locality of Reference

Caches work because of localityвҖ”programs tend to access the same data (or nearby data) repeatedly.

Types of Locality:
в•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җ

1. TEMPORAL LOCALITY (Time)
   вҖў Recently accessed data will likely be accessed again soon
   вҖў Example: Loop counter, frequently called functions
   
   for (int i = 0; i < 1000; i++) {  /* i accessed 1000 times */
       sum += array[i];               /* sum accessed 1000 times */
   }

2. SPATIAL LOCALITY (Space)
   вҖў Data near recently accessed data will likely be accessed
   вҖў Example: Array elements, struct fields, sequential code
   
   int array[1000];
   for (int i = 0; i < 1000; i++) {
       sum += array[i];   /* array[0], array[1], array[2]... */
   }
   /* Each array element is adjacent in memory */


Why Caches Exploit Locality:
в”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғ
вҖў Temporal: Keep recently accessed data in cache
вҖў Spatial: Load entire cache lines (64 bytes), not single bytes
  вҶ’ Access array[0] loads array[0..15] into cache
  вҶ’ array[1..15] are already cached!
Rule of Thumb: Programs spend 90% of time in 10% of code. A small, fast cache can satisfy most memory accesses if it holds the "hot" working set.

Cache Basics

A cache stores copies of frequently accessed memory in small, fast SRAM. Key concepts: cache lines, tags, and hit/miss.

Cache Structure

Cache Terminology:

вҖў Unit of transfer between cache and memory
вҖў Typically 64 bytes (exploits spatial locality)
вҖў Contains: Tag + Data + Status bits

TAG:
вҖў Identifies which memory address is stored
вҖў Part of memory address stored alongside data

VALID BIT:
вҖў Indicates if cache line contains valid data

DIRTY BIT:
вҖў Indicates if cached data differs from memory
вҖў (For write-back caches)

Cache Hit and Miss:
в•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җ

CACHE HIT:
CPU requests вҶ’ Check cache вҶ’ Found! вҶ’ Return data
Latency: 1-4 cycles (L1), 12 cycles (L2), 36 cycles (L3)

CACHE MISS:
CPU requests вҶ’ Check cache вҶ’ Not found вҶ’ Fetch from memory
                                       вҶ’ Store in cache
                                       вҶ’ Return data
Latency: 100+ cycles (memory access)


Types of Cache Misses (The Three C's):
в”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғ
1. COMPULSORY (Cold) Miss
   вҖў First access to data - never been in cache
   вҖў Unavoidable (except prefetching)

2. CAPACITY Miss
   вҖў Cache not big enough for working set
   вҖў Data evicted and accessed again
   вҖў Fix: Larger cache

3. CONFLICT Miss
   вҖў Multiple addresses map to same cache location
   вҖў Data evicted prematurely
   вҖў Fix: Higher associativity

Replacement Policies

When a cache set is full, which line do we evict? The replacement policy decides.

Cache Replacement Policies:
в•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җ

1. LRU (Least Recently Used)
   вҖў Evict line that hasn't been accessed longest
   вҖў Exploits temporal locality
   вҖў Expensive to implement perfectly (track all accesses)
   вҖў Common: Pseudo-LRU approximations
   
   Access order: A B C D A B E (4-way cache)
   Cache: [E, A, B, D]  вҶҗ C was LRU, evicted

2. FIFO (First In, First Out)
   вҖў Evict oldest line (first loaded)
   вҖў Simple to implement
   вҖў Doesn't consider access frequency
   вҖў Can evict frequently-used data

3. Random
   вҖў Pick random line to evict
   вҖў Surprisingly competitive!
   вҖў Avoids pathological worst-cases
   вҖў Simple hardware

4. LFU (Least Frequently Used)
   вҖў Evict line with fewest accesses
   вҖў Requires access counters
   вҖў Problem: Old but formerly-popular data never evicted
   

Modern CPUs: Pseudo-LRU or Tree-PLRU
в”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғ

       в”Ңв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”җ

Write Policies

When CPU writes data, when does it update main memory? Two main strategies.

Write-Through vs Write-Back

Write Policies:
в•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җ

вҖў Memory always has current data
вҖў Simple, no dirty bits needed

What if write is to address NOT in cache?

  Good if writing more data to same line soon

No-Write-Allocate (typical with write-through):
  Write directly to memory, don't load to cache
  Good if data won't be read soon

MESI Protocol (Most common coherence protocol):
в”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғв”Ғ
Each cache line has a state:

M (Modified): This cache has only copy, it's dirty
              вҖў Can read/write freely
              вҖў Must write back before others can read

E (Exclusive): This cache has only copy, it's clean
              вҖў Can read freely
              вҖў Can transition to Modified on write

S (Shared): Multiple caches may have copies
              вҖў Can read freely
              вҖў Must broadcast before writing (вҶ’ invalidate others)

I (Invalid): Line not valid / not present
              вҖў Must fetch from memory or other cache

Performance Impact: Cache coherence adds overhead. "False sharing" occurs when different cores write to different variables that happen to be on the same cache lineвҖ”causing constant invalidations even though there's no true sharing! 
/* False Sharing Example */
struct CountersBad {
    int counter0;  /* Core 0 increments */
    int counter1;  /* Core 1 increments */
};  /* Both on SAME cache line! Constant invalidations */

/* Fix: Pad to separate cache lines */
struct CountersGood {
    int counter0;
    char padding[60];  /* 64-byte cache line alignment */
    int counter1;
};  /* Each counter on OWN cache line */

For memory-intensive workloads, NUMA placement matters!
# Check NUMA topology on Linux
$ numactl --hardware
available: 2 nodes (0-1)
node 0 cpus: 0-7
node 0 size: 65536 MB
node 1 cpus: 8-15  
node 1 size: 65536 MB
node distances:
node   0   1 
  0:  10  21    # Local=10, Remote=21 (2.1Г— slower)
  1:  21  10

# Run process on specific NUMA node
$ numactl --cpunodebind=0 --membind=0 ./my_program

# View NUMA statistics
$ numastat
                           node0           node1
numa_hit              1234567890      987654321
numa_miss                   1234         56789  # Cross-node accesses (bad)
numa_foreign                5678          1234
NUMA Optimization: Allocate memory on the same node as the threads that use it. Linux's "first touch" policy places pages on the node where they're first accessed. Pre-fault pages in the right context!

Conclusion & Next Steps

Memory hierarchy is fundamental to computer performance. We've covered:

  • Memory Pyramid: Speed vs capacity trade-offs
  • Locality: Temporal and spatial access patterns
  • Cache Mapping: Direct, fully associative, set-associative
  • Replacement: LRU, FIFO, random policies
  • Write Policies: Write-through vs write-back
  • Coherence: MESI protocol for multicore
  • NUMA: Non-uniform memory access in multi-socket systems
Key Insight: Writing cache-friendly code means accessing memory sequentially, reusing data while it's cached, and avoiding false sharing in multithreaded programs.
